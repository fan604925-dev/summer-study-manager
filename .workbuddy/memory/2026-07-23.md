## 修复：全新打开链接积分清零（严重数据丢失风险）
- **根因**：`localStorage` 在隐私模式 / iOS 主屏幕App(standalone) / 部分 WebView 关闭后会丢失；旧 `save()` 的 `try/catch` 静默吞掉写入失败，数据只存内存 → 关闭即丢。用户"点恢复网页才保留"= 旧内存进程存活现象。
- **修复方案**：`js/store.js` 增加 IndexedDB 冗余持久化
  - `save()` 同步写 localStorage + 异步写 IndexedDB（库 `ssm-db-v1`）
  - `init()` 末尾 `_syncIdb()`：先备份内存到 IDB；localStorage 缺失时从 IDB 恢复并一次性 reload（sessionStorage 防循环）
  - `clearData()` 末尾调 save，同步清 IDB 备份
  - `_lsAvailable` 标记 localStorage 可用性，禁用时自动切换 IDB
- **验证**：`node --check` 语法通过；WebFetch 确认 GitHub Pages 已上线含 `_syncIdb`
- **部署**：commit `6be04de`，已 push 到 `fan604925-dev/summer-study-manager` main；GitHub Pages 自动重建

## 新增功能：家务劳动任务 + 零花钱兑换（v17）
- **需求**：培养孩子做家务；孩子希望积分能换钱
- **关键设计（防"把学习商品化"）**：家务积分与学习积分**分离**为两个池
  - 学习积分 `points`：原逻辑不变，兑体验/庆祝奖励
  - 家务积分 `cashablePoints`（key `ssm_cashablePoints`）：仅 `cashable:true` 的任务累加
  - `markComplete`/`markIncomplete` 按 `template.cashable` 路由到对应池；奖励 `type:'cash'` 走 `spendCashablePoints`
- **家务任务**：洗碗(5分/日)、扫地(3分/日)、拖地(3分/日)、洗衣服(8分/周)，哥哥弟弟各一套（8个模板，避免'both'导致两人共用一个实例）
- **零花钱兑换比例：1元=10分（0.1元/分）** → 每日家务≈1.1元，月零花钱≈30-40元（合理）
  - brother 零花钱奖励：5元(50分)/10元(100分)/20元(200分)
  - little 零花钱奖励：3元(30分)/5元(50分)/10元(100分)
- **UI**：`rewards.html` 拆成「可选庆祝方式(学习积分)」+「零花钱兑换(家务积分💰)」两区，弹窗对零花钱用劳动报酬话术；`tasks.html` 孩子标签显示 `⭐学习 💰家务`；兑换记录合并两池
- **DATA_VERSION 升 17** 触发老用户 `_repairData` 自动拿到新家务任务+零花钱奖励（奖励按 `childId_name` 合并新增；cashablePoints 缺失默认0）
- **测试**：`test_store.js` 新增 16.x（家务只加家务池/学习只加学习池/零花钱从家务池扣且不影响学习积分/体验奖励不扣家务积分），全部 58 项通过
- **部署**：commit 推送 main（后台进行）；push 完 GitHub Pages 自动重建
- **比例可调**：若想调，改 DEFAULT_REWARDS 里 cash 奖励的 cost 即可（父/子分档已设好）

## 2026-07-23 下午 - 修复家务积分勾选不增加（v18）
- **现象**：用户截图显示“洗碗”任务已勾选并显示 +5/5 分，但奖励页“家务积分（可换零花钱）”仍为 0
- **根因分析**：
  1. 旧 `_ensureInstancesForDate` 创建实例时未保存 `cashable` 标记，markComplete 只能依赖 `template.cashable`
  2. 用户数据里存在一个 10 分钟“洗碗”自定义模板（ likely 之前手动添加的 `childId='both'` 任务），不是 v17 默认的 cashable 模板
  3. 旧 `markComplete` 对 `childId='both'` 的任务只往 `points['both']`/`cashablePoints['both']` 写分，哥哥/弟弟各自积分不显示
- **修复内容**：
  - `DATA_VERSION` 升 18，触发 `_repairData` 自动修正
  - `_ensureInstancesForDate` 新实例/顺延实例都写入 `cashable: !!template.cashable`
  - `_repairData` 增加：自定义模板名字含“洗碗/扫地/拖地/洗衣服”的自动标记 `cashable: true`；遍历所有旧实例按当前模板同步 `cashable` 标记；`cashablePoints` 结构迁移
  - `markComplete`/`markIncomplete` 改用 `instance.cashable ?? template.cashable` 判断，并支持 `childId='both'` 时给哥哥、弟弟两个池都加/扣分
- **影响**：
  - 已勾选的 10 分钟“洗碗”如果是 both 任务，现在哥哥和弟弟都会到账（学习积分或家务积分）
  - 真正的默认 15 分钟 cashable 洗碗完成后，家务积分会正确 +5
  - 建议用户删掉重复的“洗碗”自定义任务（任务页 🗑️），只保留默认的 cashable 版本
- **文件改动**：`js/store.js`（核心修复+数据迁移）、`test_store.js`（新增 both 任务+自定义家务识别回归测试）
- **测试**：全部测试通过（61 项断言）
- **部署**：commit `3cdcd1d`，push main 成功；WebFetch 确认 `instance.cashable ?? template.cashable` 已上线
